Asynchronous Process Call Injection (Windows)

Executing shellcode within a process.

This entry aims to explain what is an Asynchronous Process Call, how does it work and how it can be exploited to execute shellcode in a Windows environment. Please note that this dwells deep on the inner workings of a computer so if you aren't familiar with processes, assembly and OS structuring, it's going to be quite hard to understand.

I. Definition of an APC +

An Asynchronous Procedure Call (APC) is a function that executes asynchronously in the context of a particular thread. On simpler terms, it's a queued code stored within a specific part of a process that runs when such part is free of any work. On even simpler terms, it's a method used by computers that allows execution without having to wait for a task to finish.

II. Lifecycle of an APC +

To fully grasp the inner workings of APC, it's key to understand it's flow, which can be divided in the following steps:

1. Invocation
The main program invokes a function which is handled to a system API.
2. Exchange
The API captures the parameters and returns control to the program which received a placeholder (handle).
3. Background Execution
The API takes care of the task in the background (async), allowing the program to continue executing lines of codes (sync).
4. Completion
Once the task is completed, the API returns a "notification" into a queue.
5. Verification
As the program completes other tasks (sync), the queue transfers the result back to it, replacing the placeholder.


III. What is an APC Injection? +

An APC Injection is a type of code injection which involves running a payload within the address space of a legitimate process, a thread to be exact. There are many variants of this type of injection: AtomBombing, Early Bird, NtQueueApcThread...

VIII. External P2 - Shellcode +

This following section can be quite dense, so I'll unpack everything carefully, please note that everything is treated with educational purposes. The idea is clear now, start a matchup in MUGEN within a thread of a process.

The goal of a shellcode is the execution of arbitrary code in a system. Scoping it to the idea, successful execution is the highest priority and for that Position Independent Code (PIC) is the best suit, since the only knowledgable aspect of the selected environment is the Operative System: Windows. PIC can be written without ASM, but you give up low-level control which in undocumented features is crucial.

The first step is finding kernel32.dll manually. In order to locate it, the code (when executed) automatically creates a Thread Environment Block (TEB), not to be confused with Thread Information Block (TIB). TEB is x64 bit and TIB is x32 bit. These blocks hold information of the currently running thread and inside them, specifically at offset 0x60, there's a pointer to the Process Environment Block which contains data about the current process. From here, there's another pointer at offset 0x18 (Ldr) which leads to the PEB_LDR_DATA structure. This structure is a record of all DLLs loaded into memory and sure enough, a list named InMemoryOrderModuleList is present at offset 0x20 which specifically represents the order of loaded modules. Inside of it, the LDR_DATA_TABLE_ENTRY awaits, providing metadata of the various modules. The thing to look for here is offset 0x20, DllBase, the base address of kernel32.dll loaded into memory. To sum things up:

Thread Environment Block -> Offset 0x60 -> Process Environment Block -> Offset 0x18 -> PEB_LDR_DATA -> Offset 0x20 -> InMemoryOrderModuleList -> Kernel32.dll -> Offset 0x20 -> DllBase

Second step, finding WinExec manually. WinExec is a deprecated function which allows for command execution, CreateProcess is a much better alternative but requires more parameters, that is why I stick to WinExec. Some DLLs contain what is known as an export table that can be inspected with tools like DllExport or Dumpbin. In such export tables we can find data such as the ordinal number (unique ID), the hint (shortened ID for backward compatibility), the Relative Virtual Address (pointer relative to the start of the loaded module) and the Name (duh):



These export tables are given by DLLs to allow other modules and programs to use them. While this feature makes modulation easier, it creates external library dependency due to the address pointing made by the Import Address Table towards the export table at runtime. There are many ways to bypass the IAT, to make it a little more interesting I have settled for manually resolving WinExec's address via GetProcAddressByHash, an alternative method of the function GetProcAddress. In essence, hash WinExec, load DllBase and PE Header data, extract information from the export table created by kernel32.dll, search the function name by hash, hash the function name, compare the hash and retrieve the function address.
